# ブログ記事ファクトチェック

指定された記事 `$ARGUMENTS` のファクトチェックを行います。

## 手順

### 1. 記事の特定と読み込み

- `$ARGUMENTS` がファイルパスならそのまま読む
- slugや部分的な名前なら `src/content/blog/` 内を Glob で検索して特定する
- 記事の本文を Read で読み込む

### 2. 記事の分析

記事を読み、以下のカテゴリに分類してファクトチェック対象を抽出する:

- **技術的な主張**: ライブラリ、フレームワーク、言語仕様に関する記述
- **数値・統計**: パフォーマンス数値、統計データ、日付
- **外部参照**: 引用元、参照先の情報
- **固有名詞**: プロジェクト名、人名、組織名のスペル

### 3. Sub Agentによる並列ファクトチェック

以下のsub agentを**並列に**起動して検証する。各agentにはチェック対象の具体的な記述を渡すこと。

#### Agent 1: 技術的正確性チェック
- 記事内の技術的な主張をWebSearchで検証
- 公式ドキュメントや信頼できるソースと照合
- API名、関数名、挙動の記述が正しいか確認

#### Agent 2: 数値・日付・統計チェック
- 記事内の数値や日付をWebSearchで検証
- リリース日、バージョン番号、統計データの正確性を確認

#### Agent 3: リンク・参照チェック
- 記事内のURLをWebFetchでアクセスし、リンク切れがないか確認
- リンク先の内容が記事の文脈と一致しているか確認

### 4. 結果レポート

全agentの結果を集約し、以下の形式でレポートする:

```
## ファクトチェック結果: [記事タイトル]

### 問題あり (要修正)
- [ ] 具体的な問題の説明と修正案

### 注意 (確認推奨)
- [ ] 確認が必要な項目の説明

### 検証済み (問題なし)
- [x] 確認できた事実の一覧
```

## 重要な注意事項

- 各sub agentには検証対象の**具体的な記述**を渡すこと(「記事全体をチェックして」ではなく「Xという主張を検証して」のように)
- 検証できなかった項目は「注意」として報告する
- 主観的な意見や感想はファクトチェックの対象外
- 記事の修正は行わず、レポートのみ出力する
