# ブログ記事の執筆

指定されたテーマ `$ARGUMENTS` に基づいてブログ記事を執筆します。

## 手順

### 1. テーマの確認と参考記事の収集

- `$ARGUMENTS` からテーマ・タイトルを把握する
- `src/content/blog/` 内の過去記事をGlob/Grepで検索し、テーマに関連する記事や近いジャンルの記事を3〜5本読み込む
- 関連する記事がない場合は、直近の記事を3本程度読み込み、文体の参考にする

### 2. 必要に応じた情報収集

- テーマに関する技術的な正確性を担保するため、WebSearchで最新情報を確認する
- 記事中で参照すべきURL（公式ドキュメント、関連記事等）がある場合はWebFetchで内容を確認する
- 事実に基づいた記述を心がけ、不確かな情報は記事に含めない

### 3. 記事の執筆

以下の文体・構成ルールに従って記事を書く。

#### frontmatter

```yaml
---
title: '記事タイトル'
description: '記事の簡潔な説明'
tag: [関連タグ]
pubDate: 'YYYY/MM/DD'
---
```

- `pubDate` は今日の日付を使用する（YYYY/MM/DD形式）
- `tag` はテーマに合った適切なタグをつける
- `draft: true` を含めて下書き状態で作成する

#### ファイル名

- 最近の記事に倣い、内容を表す英語のslug形式にする（例: `get-pr-ref-on-pr-comment.md`）
- 日付はファイル名に含めない

#### 文体のルール（過去記事から抽出）

- **一人称**: 「自分」または「僕」を使う
- **語尾**: 「〜だ」「〜と思う」「〜している」など、である調とですます調が混在するカジュアルな文体。堅くなりすぎない
- **トーン**: プロフェッショナルだがカジュアル。技術的な正確性は保ちつつ、個人の感想や考えを率直に書く
- **導入**: 短く端的に本題に入る。「書いた」「読んだ」「やった」のように一言で始めることも多い
- **括弧書き**: 補足や本音を括弧で挟むスタイル（例: 「(わざわざ言ったりしないけど)」「(ように見える)」）
- **口語的表現**: 「〜なのかな」「〜だなぁ」「〜かもしれない」等、考えながら書いている雰囲気を出す
- **外部リンク**: URLを単独行に置いてリンクカードとして表示する（`@sota1235/remark-link-bookmark`による自動変換）
- **見出し構成**: `##` を大見出し、`###` をサブセクションとして使う。`#` は基本使わない（最近の記事の傾向）
- **結び**: 「最後に」「おわりに」セクションで短くまとめる。「〜幸いです」「〜と思う」で締めることが多い
- **冗長さの回避**: 不必要に丁寧すぎる表現は避ける。簡潔に、でも思考のプロセスは残す

#### 記事タイプ別の構成パターン

**技術記事の場合:**
1. 何がしたいか / 背景
2. 方針 / アプローチ
3. 結論（コード例含む）
4. 解説
5. おわりに

**感想・レビュー記事の場合:**
1. 対象の紹介（短く）
2. 感想（つらつらと、複数の観点をサブセクションで）
3. 最後に

**振り返り・雑記の場合:**
1. 経緯・背景
2. 本題（複数の観点をサブセクションで）
3. その他（補足的な話題）
4. 最後に

### 4. ファイルの作成

- `src/content/blog/` に新しいMarkdownファイルを作成する
- 作成後、ファイルパスを報告する

### 5. 仕上げの確認

記事を読み返し、以下を確認する:

- [ ] frontmatterのフォーマットが正しいか
- [ ] 文体が過去記事と一貫しているか（丁寧すぎないか、AIっぽくないか）
- [ ] 見出しの階層が適切か
- [ ] 外部リンクが単独行に配置されているか
- [ ] 技術的な記述に明らかな誤りがないか

## 重要な注意事項

- **AIっぽい文章にしない**: 「〜について解説します」「いかがでしたでしょうか」のような定型表現は絶対に使わない
- **過度な装飾を避ける**: 絵文字は基本使わない（過去記事でもほぼ使っていない）
- **自然体を心がける**: 完璧にまとめようとせず、「うまく言語化できてない部分もあるが個人ブログなのでよしとする」くらいの気楽さで書く
- **draft: true** を必ずつけて、ユーザーが確認してからpublishできるようにする
- 記事の修正を求められた場合はEditツールで対応する
