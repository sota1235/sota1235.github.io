# 口頭メモからブログ記事の初稿を作成する

ユーザーが口頭で話した内容や、文体を意識せずに書いたメモ・テキストファイルをもとに、ブログ記事の初稿を作成します。

## 手順

### 1. 入力テキストの読み込み

- `$ARGUMENTS` がファイルパスならReadで読み込む
- ファイルが見つからない場合はGlobで検索して特定する
- 入力テキストの内容を把握し、以下を分析する:
  - **主題**: 何について話しているか
  - **主張・意見**: 筆者が伝えたいこと、感じたこと
  - **具体例・エピソード**: 話の中に含まれる具体的なエピソードや事例
  - **言及されている固有名詞**: ツール名、書籍名、サービス名、人名等
  - **記事タイプの判定**: 技術記事 / 感想・レビュー / 振り返り・雑記 のどれに近いか

### 2. 過去記事の参照

- 入力テキストのテーマに関連するキーワードで `src/content/blog/` 内をGrepし、近いジャンルの過去記事を3〜5本読み込む
- 関連する記事がない場合は、直近の記事を3本程度読み込む
- 読み込んだ記事から、そのジャンルでの著者の書き方（構成の組み立て方、どの程度深掘りするか、どういう切り口で語るか）を把握する

### 3. 必要に応じた情報補完

- 入力テキスト内で言及されているURL、ツール名、書籍名などについてWebSearchで正確な情報を確認する
- 正式名称、URLなどを補完する（入力テキストでは口頭ベースなので曖昧なことがある）
- ただし、入力テキストに含まれていない主張や意見を勝手に追加しない

### 4. 記事の構成設計

入力テキストの内容を整理し、記事としての構成を設計する。

- 入力テキストの話の流れを尊重しつつ、読みやすい順序に再構成する
- 口頭ならではの繰り返しや脱線は整理するが、**著者の思考の流れや脱線気味の補足は「著者らしさ」として残す**
- 必要に応じてセクション分けを行う

#### 記事タイプ別の構成パターン

**技術記事の場合:**
1. 何がしたいか / 背景
2. 方針 / アプローチ
3. 結論（コード例含む）
4. 解説
5. おわりに

**感想・レビュー記事の場合:**
1. 対象の紹介（短く）
2. 感想（つらつらと、複数の観点をサブセクションで）
3. 最後に

**振り返り・雑記の場合:**
1. 経緯・背景
2. 本題（複数の観点をサブセクションで）
3. その他（補足的な話題）
4. 最後に

### 5. 記事の執筆

以下の文体・構成ルールに従って、入力テキストの内容をブログ記事に仕上げる。

#### frontmatter

```yaml
---
title: '記事タイトル'
description: '記事の簡潔な説明'
tag: [関連タグ]
pubDate: 'YYYY/MM/DD'
draft: true
---
```

- `pubDate` は今日の日付を使用する（YYYY/MM/DD形式）
- `tag` はテーマに合った適切なタグをつける
- **`draft: true` を必ず含める**

#### ファイル名

- 内容を表す英語のslug形式にする（例: `get-pr-ref-on-pr-comment.md`）
- 日付はファイル名に含めない

#### 文体のルール（過去記事から抽出）

- **一人称**: 「自分」または「僕」を使う
- **語尾**: 「〜だ」「〜と思う」「〜している」など、である調とですます調が混在するカジュアルな文体。堅くなりすぎない
- **トーン**: プロフェッショナルだがカジュアル。技術的な正確性は保ちつつ、個人の感想や考えを率直に書く
- **導入**: 短く端的に本題に入る。「書いた」「読んだ」「やった」のように一言で始めることも多い
- **括弧書き**: 補足や本音を括弧で挟むスタイル（例: 「(わざわざ言ったりしないけど)」「(ように見える)」）
- **口語的表現**: 「〜なのかな」「〜だなぁ」「〜かもしれない」等、考えながら書いている雰囲気を出す
- **外部リンク**: URLを単独行に置いてリンクカードとして表示する（`@sota1235/remark-link-bookmark`による自動変換）
- **見出し構成**: `##` を大見出し、`###` をサブセクションとして使う。`#` は基本使わない
- **結び**: 「最後に」「おわりに」セクションで短くまとめる。「〜幸いです」「〜と思う」で締めることが多い
- **冗長さの回避**: 不必要に丁寧すぎる表現は避ける。簡潔に、でも思考のプロセスは残す

#### 入力テキストからの変換で特に意識すること

- **口頭の「えーと」「なんか」等のフィラーは除去する**が、「〜な気がする」「〜かもしれない」等の思考の揺れは文体として活かす
- **口頭での順序そのままにしない**: 話し言葉は往々にして結論が後回しになるので、記事として読みやすい順序に再構成する
- **口頭で省略された文脈を補う**: 話し言葉では「あれ」「それ」で通じる指示語を、読者にわかるよう具体化する
- **口頭の熱量やニュアンスは保つ**: 「これめっちゃ良かった」→ そのままでもいいし、文脈に合わせて残す。過度に丁寧にしない
- **入力テキストにない内容を捏造しない**: 著者が言っていないことを勝手に足さない。構成の都合で補足が必要な場合は `<!-- TODO: ここの詳細を補足する -->` のようにコメントで残す

### 6. ファイルの作成

- `src/content/blog/` に新しいMarkdownファイルを作成する
- 作成後、ファイルパスを報告する

### 7. 仕上げの確認

記事を読み返し、以下を確認する:

- [ ] frontmatterのフォーマットが正しいか（`draft: true` が含まれているか）
- [ ] 入力テキストの主張や意見が漏れなく反映されているか
- [ ] 入力テキストにない内容が勝手に追加されていないか
- [ ] 文体が過去記事と一貫しているか（丁寧すぎないか、AIっぽくないか）
- [ ] 見出しの階層が適切か
- [ ] 外部リンクが単独行に配置されているか

確認後、以下を報告する:
- 作成したファイルのパス
- 入力テキストから記事にする際に判断に迷った点や、補足が必要と思われる箇所
- `<!-- TODO -->` コメントを入れた箇所があればその一覧

## 重要な注意事項

- **入力テキストの内容を尊重する**: これが最優先。著者の主張、意見、感想を忠実に記事化する。AIが勝手に意見を追加したり、主張を変えたりしない
- **AIっぽい文章にしない**: 「〜について解説します」「いかがでしたでしょうか」「それでは見ていきましょう」のような定型表現は絶対に使わない
- **過度な装飾を避ける**: 絵文字は基本使わない（過去記事でもほぼ使っていない）
- **自然体を心がける**: 完璧にまとめようとせず、「うまく言語化できてない部分もあるが個人ブログなのでよしとする」くらいの気楽さで書く
- **不明点は補完せず残す**: 入力テキストで曖昧な部分を勝手に解釈せず、`<!-- TODO -->` コメントで著者に確認を促す
- 記事の修正を求められた場合はEditツールで対応する
